Домашняя BigData. Часть 2. Практика MLlib на кластере AWS

  • Big Data
  • Tutorial
Здравствуйте.

В данной статье мы в домашних условиях произведем установку 3-х узлового кластера Hadoop Cloudera на платформу EC2 AWS (Amazon Web Services) и воспользуемся библиотеками MLlib из Apache Spark для задач с машинным обучением.



Авторизация на платформе Amazon Web Services



Предполагаю, что вы ознакомились с первой частью и успешно установили Apache Spark на платформу, теперь для хостинга кластера мы создадим ещё 3 узла на базе решений AWS (Amazon Web Services).

Если же у вас ещё нет опыта работы с EC2 AWS, то не переживайте. Вы научитесь развертывать кластер Hadoop с нуля в этой статье.

1.1. По ссылке aws.amazon.com/console вам предстоит зарегистрироваться. Введите имя и запомните пароль.

2.1 Выберите в меню «Services->EC2». Для начала использования сервиса EC2 вам нужно запустить экземпляр виртуального сервиса (EC2 instance).

2.2 Далее необходимо выбрать версию операционной системы имиджа виртуальной машины (AMI- Amazon Machine Image), выбираем Red Hat Enterprise Linux 7.5 (HVM), SSD Volume Type — ami-28e07e50, жмем ”Select". Переходим к настройке экземпляра сервера:

Примечание: для свободного использования ищем значок «Free tier eligible».

• Выбираем тип «t2.medium» с параметрами 2vCPU, 4 GB памяти, General Purpose. Жмем «Next: Configuring Instance Details».
• Добавляем количество экземпляров до 2, жмем «Next: Add Storage».
• Изменяем размер диска до 30 GB и тип на Magnetic.
• В разделе «Tag Instances» нажимаем Add Tag вводим «Habr» в поле Key и Ваше_имя (например Yuriy) в поле Value нажимаем на «Next:...»

• В разделе " Configure Security Groups " выберите опцию «Select existing security group», выбрав имя группы безопасности («ClouderaManagedNodes») и нажмите «Next ...»

• Просмотрите экран «Review» для проверки введенных значений и запустите «Launch»
• Для подключения к узлам кластера необходимо создать пару открытых ключей для идентификации и авторизации. Для этого выберите в списке тип операции «Choose an existing key pair» и выберите имя для ключа в формате имени Hadoop01. После этого запустите экземпляры узлов.

Создание ключей



• Скачиваем Putty для клиента и устанавливаем его на систему www.chiark.greenend.org.uk/~sgtatham/putty/latest.html

• Файл ключа HadoopUser01.pem использует старый формат для удобства конвертируем его в формат ppk используемый Putty. Для этого запускаем утилиту PuTTYgen, загружаем сохраненный на шаге 3.3 ключ в старом формате .pem в утилиту. Конвертируем ключ и сохраняем (Save Private Key) для последующего использования в папку c:\Hadoop с расширением .ppk

Настройка под себя



2.3 Повторите процедуру описанную в предыдущем пункте, но для узла в конфигурации t2. large/ 2vCPU/ 8 GB RAM/ 50GB Magnetic HDD, выбираем Red Hat Enterprise Linux 7.5 (HVM), SSD Volume Type — ami-28e07e50, жмем ”Select". Переходим к настройке экземпляра сервера.

2.4 Для удобства работы переименуйте узлы кластера в нотации Node01-03. Для подключения к узлам кластера с локального компьютера через SSH необходимо определить IP адрес узла и его public/private DNS имя, выберите поочередно каждый из узлов кластера и для выбранного экземпляра (instance) запишите его public/private DNS имя для подключения через SSH и для установки Cloudera Manager в текстовый файл HadoopAdm01.txt

Пример: ec2-35-162-169-76.us-west-2.compute.amazonaws.com.

Установка Cloudera Distributed Hadoop на кластер



3.1 Для установки Cloudera manager выбираем Node01 (копируем его Public DNS) для подключения через SSH. Настраиваем подключение через SSH. Используем сохраненное имя первого узла для настройки подключения через SSH с использованием пары ключей Private/Public «HadoopUser01.ppk” созданных в пункте 3.3. Переходим в раздел Connection/Auth через кнопку Browse ищем папку c:\Hadoop где мы предварительно сохранили файл „HadoopAdm01.ppk“

3.2 Подключаемся к узлу Node01 и используем login: ec2-user.

3.3 Для подключения между узлами кластера необходимо установить SSH в узлы и настроить аутентификацию узлов кластера без необходимости ввода пароля при подключении узлов друг с другом, для этого:

• В консоли проверяем наличие SSH на Node01 пытаясь установить соединение с Node02
ssh ec2-user@ip-172-31-32-131.us-west-2.compute.internal.

# подтвердите занесение узла в список доверенных (known hosts)


• Создайте файл .ssh/id_rsa для внесения в него приватного ключа из вашего файла Hadoop0X.pem.

• Просмотрите разрешения на каталог .ssh.

• Cоздаем файл в редакторе VI и копируем в него приватный ключ Hadoop0X.pem
vi .ssh/id_rsa


• Сравниваем разрешения и пытаемся установить доступ по ssh снова
вы получили «первое предупреждение, не отчаивайтесь :-)».

• Исправьте разрешения на файл .ssh/id_rsa на read только для eс2-user.

• Проверяем подключение по ssh между узлами кластера без ввода пароля.
ssh ec2-user@ip-172-31-32-131.us-west-2.compute.internal
ssh ec2-user@ ip-172-31-37-44.us-west-2.compute.internal


3.4 Устанавливаем совместимую версию Java для вашего кластера.

В свете возможного введения платы на использование Oracle JDK устанавливаем OpenJDK.
sudo yum install java-1.8.0-openjdk


Скачиваем „последнюю версии“ JDK с сайта Oracle (1.8_162) и копируем ее с помощью WinSCP на Master узел.

• Скопируем файл JDK с Master узла на оба узла с использованием команды scp.

scp jdk-8u60-linux-x64.gz ip-172-31-13-70.us-west-2.compute.internal:/home/ec2-user/
scp jdk-8u60-linux-x64.gz ip-172-31-11-33.us-west-2.compute.internal:/home/ec2-user/


Убедитесь что файлы скопированы на узлы и лежат в каталоге /home/ec2-user.

• Разархивируем JDK файл в формате tar.gz с помощью команды на всех узлах и переместим в директорию /usr/java.

tar -xvf jdk-8u162-linux-x64.tar.gz


sudo mkdir -p /usr/java
sudo mv ./jdk1.8.0_162 /usr/java/


• Теперь мы перенаправим исполняемые файлы из каталога /usr/bin/javac на новый каталог /usr/java/jdk1.8.0_162/bin/javaws с помощью команды update-alternatives.

sudo update-alternatives --install "/usr/bin/java" "java" "/usr/java/jdk1.8.0_162/bin/java" 1
sudo update-alternatives --install "/usr/bin/javac" "javac" "/usr/java/jdk1.8.0_162/bin/javac" 1
sudo update-alternatives --install "/usr/bin/javaws" "javaws" "/usr/java/jdk1.8.0_162/bin/javaws" 1


• Настроим разрешения для каталогов с исполняемыми файлами Java.
sudo chmod a+x /usr/bin/java
sudo chmod a+x /usr/bin/javac
sudo chmod a+x /usr/bin/javaws
sudo chown -R root:root /usr/java/jdk1.8.0_162


• Обновим переменные окружения на всех узлах для обновленных исполняемых файлов Java.
--Set JAVA_HOME (on all nodes) для bash
# создаем файл скрипта для модификации bash среды для каждого профиля
sudo vi /etc/profile.d/java.sh
# вносим туда нижележащую запись и сохраняемся
JAVA_HOME=/usr/java/jdk1.8.0_162
# для верности сделайте выход из консоли для проверки результата


3.5 После установки JDK на всех узлах нам необходимо оптимизировать работу файловой системы, ядра и памяти для узлов Hadoop.
• Выполните проверки следующих параметров на всех узлах кластера.
• Отключаем sestatus Security Enhanced Linux на всех узлах access.redhat.com/documentation/en-us/red_hat_enterprise_linux/6/html/security-enhanced_linux/sect-security-enhanced_linux-enabling_and_disabling_selinux-disabling_selinux
# редактируем файл /etc/selinux/config
sudo vi /etc/selinux/config

SELINUX=disabled
# требует перезагрузки


• Оптимизируем работу памяти для виртуальных машин.

access.redhat.com/documentation/en-us/red_hat_enterprise_linux/7/html/performance_tuning_guide/sect-red_hat_enterprise_linux-performance_tuning_guide-configuration_tools-configuring_system_memory_capacity

# редактируем файл
sudo vi /etc/sysctl.conf

vm.overcommit_memory = 1
vm.swappiness = 1
vm.overcommit_ratio = 50
#Применение настроек из conf файла

sysctl -p

#Очистка своп файла без перезагрузки

swapoff -a


• Отключаем использование Transparent Huge Pages
access.redhat.com/documentation/en-us/red_hat_enterprise_linux/6/html/performance_tuning_guide/s-memory-transhuge
access.redhat.com/documentation/en-us/red_hat_enterprise_linux/7/html/virtualization_tuning_and_optimization_guide/sect-virtualization_tuning_optimization_guide-memory-tuning

#add lines to init scripts

vi /etc/rc.d/rc.local

echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag
# делаем его исполняемым
chmod +x /etc/rc.d/rc.local

# Add the following line to the GRUB_CMDLINE_LINUX options in the /etc/default/grub file

vi /etc/default/grub
# добавьте в строку GRUB_CMDLINE_LINUX=
transparent_hugepage=never (спросите куда у препода)
# модифицируем имидж

grub2-mkconfig -o /boot/grub2/grub.cfg


На всякий случаем перегружаем системы.

3.6 После установки JDK на всех узлах мы готовы к загрузке всех необходимых Hadoop компонентов Cloudera.

• Устанавливаем wget для загрузки файлов.

sudo yum install wget


• Устанавливаем локальный репозиторий дистрибутива Cloudera Distributed Hadoop и Cloudera Manager.

sudo wget https://archive.cloudera.com/cm5/redhat/7/x86_64/cm/cloudera-manager.repo -P /etc/yum.repos.d/


• Редактируем версию репозитория до 5.12.1 или другой желаемой версии
sudo rpm --import https://archive.cloudera.com/cdh5/redhat/7/x86_64/cdh/RPM-GPG-KEY-cloudera


• Скачиваем инсталлятор Cloudera Manager нужной версии.
wget https://archive.cloudera.com/cm5/installer/5.12.1/cloudera-manager-installer.bin

# делаем его исполняемым
chmod u+x cloudera-manager-installer.bin

# запускаем установку из локального репозитория
sudo ./cloudera-manager-installer.bin --skip_repo_package=1

# или интернета

sudo ./cloudera-manager-installer.bin


3.7 Запускаем инсталляцию и устанавливаем Cloudera Manager на первый узел кластера. По окончанию установки веб интерфейс будет доступен по адресу: localhost:7180 со стандартным логином и паролем admin/admin.

Для доступа в пользовательскому интерфейсу замените localhost в линке из пункта 3.4 на publicDNS имя первого узла на котором установлен Cloudera Manager.

Пример: ec2-35-162-169-76.us-west-2.compute.amazonaws.com:7180.

3.8 Завершите установку Cloudera Manager используя логин/пароль admin/admin. Примите лицензионное соглашение и укажите оставшиеся узлы для установки CDHadoop.

• Для поля „Specify Hosts for your CDH cluster installation“ введите оставшиеся 3 узла кластера для установки (внутренние DNS имена) через запятую и нажмите кнопку Search.

• Проверьте что все обнаруженные узлы не являются частью других кластеров и доступны. Выберите Continue.

• В окне Select Repository подтвердите опции по умолчанию (последняя версия Cloudera Manager, использовать установку Parcels, не устанавливать другие пакеты).

• В окне JDK Installation Options примите лицензионное соглашение и установку политики шифрования для Java (Kerberos).

• В окне Enable Single User Mode не активизируйте данную опцию. Она несколько упрощает вариант установки, но в нашем варианте не подходит.

Попейте чайку, вздохните свежего воздуха и продолжаем



3.9 Учитывая выбор не Single User Mode нам придется настраивать логин и пароль для запуска и установки кластера на каждом узле. Выбираем non root учетную запись (ec2-user) и аутентификацию по приватному ключу Hadoop01.ppk который мы использовали ранее.

• Первоначально мы ставим только Cloudera Agent и JDK на каждый узел кластера и потом устанавливаем выбранные парселы (distribution, unpacking, activating). По завершении инсталляции Cloudera Manager проверяет все установленные парселы, корректность установок и опций.

• Вы можете получить уведомление о не оптимальной настройке узлов вашего кластера связанно это в минимальной конфигурацией узлов (размер ОЗУ 1Гб) предполагающей свопинг. Игнорируйте данное сообщение.

• Завершаем инсталляцию выбираем Core Hadoop Services. В окне Customized Role Assignments выбираем место для установки Ролей для оставшихся узлов кластера. Оставьте по умолчанию, либо обратитесь к интернету по вопросу выбора совместимых компонент для размещения на узлах.

• Завершаем установку компонент на узлы кластера Hadoop. После успешного завершения установки вы должны получить работающий кластер с небольшими конфигурационными замечаниями.

• Устраните по возможности замечания по каждому из компонентов Hadoop и проверьте работоспособность работы HDFS и основных компонент.

• Подключитесь к первому узлу кластера Node01.

• Проверьте конфигурационные файлы для кластера и всех компонент.

cd /etc/hadoop/conf
ls -ll


• Найти файлы установки при использовании парселов можно по адресу.

cd /opt/cloudera/parcels


• Перед тем как проверить работоспособность файловой системы HDFS необходимо создать домашний каталог пользователя и предоставить доступ к нему доступ для пользователя ec2-user.
sudo -u hdfs hadoop fs -mkdir /user
sudo -u hdfs hadoop fs -mkdir /user/ec2-user
sudo -u hdfs hadoop fs -chown ec2-user:ec2-user /user/ec2-user


• Создадим тестовый текстовый файл с помощью редактора VI и скопируем данный файл в HDFS.
vi hdfs-test
hadoop fs -copyFromLocal hdfs-test /user/ec2-user
hadoop fs -ls


• Проверим работоспособность YARN для этого используем программу встроенную в дистрибутив для оценки ПИ по методу Монте-Карло
hadoop jar /opt/cloudera/parcels/CDH/lib/hadoop-0.20-mapreduce/hadoop-examples.jar. pi 20 50
